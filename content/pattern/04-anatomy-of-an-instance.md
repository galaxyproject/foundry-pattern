# Anatomy of an Instance

A Foundry is an inspectable, human-readable knowledge base whose *structure* makes it executable: its knowledge is decomposed into typed units that compile into frozen artifacts carrying provenance, and the authored knowledge cannot reach a trusted result except through a gate it is forbidden to skip. That sentence names everything a Foundry has. This page separates the parts that are the *same* in every instance — the invariants — from the parts each instance gets to *choose* — the variables. The headline variable is the gate.

Where [[the-diff]] earns the abstraction comparatively (hold the two instances side by side; what survives the diff is real), this page earns it descriptively: walk the anatomy once, and mark each part invariant or variable. For the underlying model and term definitions, see [[the-model]] and the [[glossary]].

## The invariants

These are the same in every instance. Remove any one and the thing stops being a Foundry — so each is stated with what breaks without it.

**The knowledge base / reader's surface.** The source of truth is authored to be *read and learned by a human*, not merely stored for an agent to retrieve. It renders as a navigable site with links, backlinks, and progressive disclosure. Without it, the knowledge is only ever machine feed — you lose the human's ability to scrutinize, correct, and trust the source, and the project collapses into the very skills-as-source pattern the Foundry exists to invert.

**The Mold (typed reference manifest).** The unit of the KB is an abstract action description whose frontmatter *declares* the references it depends on — other KB pages, schemas, CLI manual pages, prompts, examples — by type. The Mold is a source artifact, independent of any runtime. Without typed structure, "make the KB executable" has nothing to grab: a prose page can be read but not compiled, resolved, or drift-checked. The types are what let the compiler dispatch.

**The Cast (deterministic-first, LLM-second compilation).** Casting turns a Mold into a target-specific skill artifact, doing as much as possible *deterministically* (verbatim copies, sidecars, schema serialization) and reserving the LLM for the few kinds where condensation adds value — in that order of trust. Casting is the **integration boundary**: artifacts come out condensed, isolated, and frozen, with no links back to the KB. Without the deterministic-first discipline, you cannot reproduce or diff a cast, and "compiled" becomes a synonym for "regenerated by a model each time" — untraceable by construction.

**Provenance.** Every cast emits a record beside it: which Mold revision, which model and prompt version, which references resolved, which checks ran. Without provenance you cannot tell a real-and-verified claim from a plausibly-confabulated one, drift is undetectable, and the corpus becomes model output fed back as ground truth — the degenerative loop the pattern is built to break.

**The Pipeline.** Molds compose into ordered end-to-end tasks. Without composition, a Foundry is a bag of disconnected actions with no journey through them — and, crucially, no altitude at which a gate can live (see below).

**Corpus-grounding.** Abstractions trace back to real exemplars in an external corpus, cited by URL/reference and not mirrored in. Without grounding, the KB is invented top-down — restated canonical knowledge with no anchor, exactly the commodity a frontier model already regenerates on demand.

**Progressive disclosure.** Pipelines disclose the journey, Molds the action, typed references the dependency surface — each level revealed only when needed. Without it the reader's surface drowns and the runtime over-loads context; disclosure is what keeps both the human and the agent oriented.

## The variables

These are chosen per instance. Two are mundane; one is the whole game.

**The domain and corpus.** *What* knowledge the Foundry curates, and *which* external corpus grounds it. One instance grounds in a curated workflow corpus; another in established-methods literature plus cautionary negative examples. The grounding *discipline* is invariant; the corpus is the variable.

**The target format(s).** *What* a cast produces — one runtime's skill format, a generic bundle, a web payload. The KB stays the source of truth; the target is one output, and an instance may have several. The integration boundary is invariant; which side(s) it compiles toward is the variable.

**The gate.** This is the headline. **The gate is the external, non-self-certifying check that stands between authored knowledge and a trusted artifact or result.** Every Foundry has one — knowing is not gating; a model may *mention* a caveat, but only a gate makes *doing not terminate* until something external clears it. What the gate *is* differs per instance, and that is the axis on which the pattern's abstraction is earned.

Think of the gate as a **slot**: invariant in that it must be filled, variable in what fills it. Two realizations, precisely:

- **A deterministic CLI gate.** A parser/validator decides, mechanically, whether the authored artifact is well-formed. It is fast and *total over what it covers* — every failure mode within its reach is caught the same way every time, no prose caveats — but it only covers what a parser can decide. (In [[galaxy-workflow-foundry]]: `gxwf` parses and validates the workflow format and tool steps; hallucinated identifiers, dropped revisions, and bad state are caught at compile-time, not narrated.)

- **An empirical referee gate.** When no parser *can* decide validity — "is this method sound?" has no mechanical oracle — the gate is itself *authored knowledge cast into a skill*: an `analyze → referee → revise` loop in which doing hands off to a referee whose verdict gates certification. The referee both **critiques** (reasons about known invalidity patterns) and **calibrates** — it partly *constructs its own check*, building and running the empirical test the field trusts (permutation under the null, simulation against known truth, calibration). "Doing never self-certifies": reasoning is necessary but not sufficient, so the strong gate requires at least one constructed, external check. (This is the spine of [[statistical-genomics-foundry]].)

The generalization: *a deterministic CLI is one kind of gate; an empirical referee Mold is another.* A CLI is pure infrastructure you call; a referee is a deliverable you produce — the gate becomes a Mold cast into a skill. The slot is invariant; the filler varies along a clean axis from "mechanical and total-but-narrow" to "constructed and broad-but-non-total."

Note the **altitude**: the gate lives at protocol/pipeline level, not as a property hung on a single Mold. It is the obligation that a *doing* pipeline may not reach certification except through the check — encoded as a pipeline/protocol convention (a terminal validation or `[gate]`-style phase), reusing the same composition spine both instances already have.

## Invariant vs. variable, at a glance

| The slot (invariant in every instance) | The filler (chosen per instance) |
|---|---|
| Reader-facing KB / progressive disclosure | Which domain's knowledge |
| Mold = typed reference manifest | — (shape fixed; content domain-specific) |
| Cast = deterministic-first integration boundary | Which target format(s) |
| Provenance beside every artifact | — (always required) |
| Pipeline composition | — (always required) |
| Corpus-grounding | Which external corpus |
| **The gate** | **deterministic CLI** *(parser)* — or — **empirical referee Mold** *(analyze→referee→revise, partly self-constructed)* |

## How to read this

If you are building a third Foundry, the invariants are *given* — you inherit the KB-as-source-of-truth, the typed Mold, deterministic-first casting, provenance, composition, grounding, and disclosure essentially for free (that inheritance is what makes deriving from the pattern cheaper than starting fresh). Your domain and target format are routine choices. The real design work — the one decision the abstraction hands you and refuses to make for you — is **filling the gate slot**: what is the external, non-self-certifying check between your authored knowledge and a trusted result, and does a parser already decide it or must you *construct* the check as authored knowledge cast into a skill? Get the gate right and the rest of the anatomy is scaffolding. See [[guiding-principles]] for the commitments that hold across whichever gate you choose.
